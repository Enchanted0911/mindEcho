# README 架构文档维护专家

负责将整个项目的设计集中维护在 **README.md** 中，并以此作为后续需求开发与设计的唯一指引。

## 格式与规范约束

- **README 结构必须遵循**：[docs/README-template.md](docs/README-template.md)。更新或新增章节时，按模板中的章节顺序与标题约定填写，确保「整体系统架构 → 各模块划分 → 领域模型 → DB 信息 → 关键设计特性 → 系统集成 → 快速开始 → 监控与告警 → 常见问题 → 后续维护内容」结构一致。
- **增量设计时的强制约束**：必须参考 **specrules/rules/index.md**。根据本次变更所涉及的任务类型（架构与分层、设计文档与领域建模、DB/持久化、MQ 等），从 `specrules/rules/index.md` §2 找到对应「任务类型 → 规则集合」，**按引用路径加载对应规则**，在更新 README 或给出设计结论时遵守这些规则。不得在未加载规则的前提下做架构或设计层面的增改。

## 维护范围

README.md 中必须包含并持续维护（与模板章节对应）：

1. **整体系统架构**：分层、模块划分、技术栈、部署与运行形态
2. **DB 信息**：表结构、库/表命名、关键字段说明、索引与约束；若当前文档中无 DB 信息，**主动向用户索取**并补充
3. **各模块划分**：模块职责、边界、依赖关系、与分层架构的对应
4. **系统领域模型**：核心实体、聚合、领域概念及关系（可用文字或简图说明）
5. **后续维护内容**：版本历史、待办、已知约束、与上下游系统的集成说明等

## 工作流程

### 情况 A：用户未提供额外输入（无设计文档、无 PRD）

- **视为**：基于当前系统架构进行文档更新。
- **动作**：先读取 **docs/README-template.md** 与当前 **README.md**、项目内相关规范（如 `specrules/00_general/project_struct.md`），根据当前代码与结构，对 README 做一致性更新（按模板补全缺失章节、修正过时描述、补充 DB 等）。若发现缺少 DB 信息，明确提示用户提供。更新时保持与模板的章节顺序与标题一致。

### 情况 B：用户提供了设计文档、PRD 或明确需求描述（增量设计）

- **步骤 0（强制）**：根据本次变更涉及的任务类型，从 **specrules/rules/index.md** 加载对应规则集合（引用路径，需要时读取具体规范），作为本次增量设计的约束。
- **步骤 1**：理解用户需求（目标、范围、变更点）。
- **步骤 2**：结合当前 README.md、**docs/README-template.md** 与实际项目结构，识别与现有架构的差异与影响点。
- **步骤 3**：列出**需要用户确认或澄清的点**（例如：新表归属库、模块归属、与现有领域模型的关系、配置来源等），等待用户确认。
- **步骤 4**：用户澄清后，先按 **docs/README-template.md** 的格式更新 README.md 中的架构（系统架构、模块划分、领域模型、DB 等），再以更新后的架构作为指引，继续开发或设计任务说明。所有增改须符合 **specrules/rules/index.md** 中已加载的规范。

## 需求/开发时的使用方式

- **每次做需求设计或开发前**：先从 README.md 中查找架构相关信息，确保方案与现有架构一致。
- **有增量需求时**：先按 specrules/rules/index.md 加载相关规范，再更新 README 中的架构与相关章节，再以架构为指引进行开发或拆解任务；不先改代码再补文档。

## 规范与约束

- 所有架构级结论以 README.md 为准；与 README 冲突的零散文档以 README 为权威。
- DB 信息缺失时，不猜测表名或字段，必须向用户询问并注明「待用户提供」直至补充完整。
- 更新文档时**必须**按 **docs/README-template.md** 的章节与结构执行，保持结构清晰。
- 版本历史、变更记录在 README 的「后续维护内容」中更新。
- 增量设计时**必须**引用并遵守 **specrules/rules/index.md** 中与任务类型对应的规则，不得跳过。

## 输出要求

- 更新 README 时给出具体修改段落或章节，必要时直接给出可替换的 Markdown 片段；新增章节时对齐模板中的标题与顺序。
- 需要用户确认时，用列表明确列出问题，并说明每个问题对架构/文档的影响。
- 澄清完成后，先总结「本次架构变更点」，再给出 README 的更新内容或修改建议。
- 若本次动用了 specrules/rules/index.md 中的某类规则，可在输出中简要注明「已按 specrules/rules/index.md 中 xxx 规范约束」。

## 自更新说明（维护本 skill 时）

本 skill **独立运行**，不依赖 `.cursor/agents/` 下任何 agent。当**项目约定**发生变化时，应更新本 SKILL.md 以保持一致：

- **docs/README-template.md** 章节或标题变更 → 更新本 skill 中「格式与规范约束」「维护范围」及工作流程里对模板结构的描述。
- **specrules/rules/index.md** 任务类型或规则路径变更 → 更新「格式与规范约束」「工作流程 步骤 0」中对 rules 的引用说明。
- **specrules/** 目录或项目结构变更 → 更新「工作流程 情况 A」中提到的规范路径（如 `project_struct.md`）。

更新后自检：本 skill 内所有 `docs/README-template.md`、`specrules/rules/index.md`、`specrules/` 的路径是否仍正确；SKILL.md 建议 ≤ 500 行，超长时可把示例挪到 [reference.md](reference.md)。
