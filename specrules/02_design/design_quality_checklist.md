# 设计质量检查清单

## 背景

本规范提供设计文档的质量检查清单,供 AI Agent 在设计完成后自动验证,或供开发者在代码审查时参考。所有检查项分为"必须"和"
推荐"两个等级,确保设计文档的完整性、正确性和可实施性。

---

## 一、整体结构检查

### 必须项

- [ ] **文档结构完整**：包含所有必填章节
- [ ] **章节编号规范**：章节编号连续,无遗漏
- [ ] **Markdown 格式正确**：标题层级、列表、代码块、表格格式规范
- [ ] **相关规范引用**：包含所有适用的 `specrules/` 规范链接

### 推荐项

- [ ] **目录清晰**：包含文档目录,便于快速导航
- [ ] **版本标记**：包含文档版本号和修订历史
- [ ] **附录完整**：术语表、缩略语、参考文档等

---

## 二、需求分析检查

### 必须项

- [ ] **背景明确**：清楚说明为什么要做这个需求
- [ ] **业务目标明确**：描述需求要解决的业务问题
- [ ] **功能范围明确**：清晰列出本期要实现的功能和不实现的功能
- [ ] **用户故事完整**：包含用户角色、场景、操作和预期结果

### 推荐项

- [ ] **业务流程图**：绘制核心业务流程图(推荐使用 Mermaid)
- [ ] **用例图**：绘制用例图,明确参与角色和系统边界
- [ ] **竞品分析**：如有类似产品,进行对比分析

---

## 三、前置评估检查(第2章)

### 必须项：业务&产品关注点表格

**表格必须包含以下评估项**：

- [ ] **功能的地区差异化控制**：by region/cityId 是否有差异
- [ ] **场景**：外卖、打赏、券包等
- [ ] **配送方式**：配送、自取
- [ ] **支付**：普通支付、钱包支付等
- [ ] **是否涉及资金**：如涉及,需引入结算评估
- [ ] **敏感合规治理**：手机号、邮箱、地址等敏感信息

**评估结果必须明确**：

- [ ] 每个评估项都有明确的"本次需求"列内容
- [ ] 涉及资金的需求必须有结算评估链接或说明
- [ ] 涉及敏感信息的需求必须有合规治理方案

### 必须项：技术关注点表格

**表格必须包含以下评估项**：

- [ ] **多机房配置**：非必要不引入差异
- [ ] **终端版本控制**：后端协议 vs 前端版本
- [ ] **接口变更**：新增下游接口依赖
- [ ] **存储变更**：MySQL、缓存、Kafka 等
- [ ] **「三高」保证**：并发安全、性能、一致性

**评估结果必须明确**：

- [ ] 勾选了所有相关的技术关注点
- [ ] 每个关注点都有"说明&要求"栏的具体内容
- [ ] 对于多机房配置,明确说明是否引入差异及原因

---

## 四、整体架构设计检查(第3章)

### 必须项

- [ ] **系统交互图**：绘制系统间的交互关系图
- [ ] **数据流向图**：明确数据如何在系统间流转
- [ ] **部署架构图**：说明系统的部署方式(如单机房/多机房)
- [ ] **技术选型说明**：列出关键技术选型及理由
- [ ] **系统间解耦（宪章原则 VIII）**：禁止系统间互相调用（禁止 A 调 B 且 B 调 A 的双向 RPC/HTTP）；允许单向调用；若需双向交互，采用单向
  RPC + MQ（例如 A→B 用 RPC，B→A 用 MQ）；架构图/时序图中不得出现 A↔B 双向同步调用

### 推荐项

- [ ] **架构决策记录(ADR)**：记录重要的架构决策及理由
- [ ] **容量规划**：评估系统容量需求(QPS、存储、带宽等)
- [ ] **扩展性设计**：说明系统的横向/纵向扩展方案

---

## 五、详细方案检查(第4章)

### 4.1 干系方检查(必填)

- [ ] **干系方表格完整**：包含业务人员、产品运营、研发、数据组等
- [ ] **影响流程明确**：每个干系方的影响流程已描述
- [ ] **SOP建议明确**：是否需要 SOP 已标注
- [ ] **干系人联系方式**：关键干系人的联系方式(或团队名称)已填写

### 4.2 业务流程图检查

- [ ] **正常流程完整**：主流程清晰可读
- [ ] **异常流程完整**：包含主要的异常处理分支
- [ ] **流程图规范**：使用 Mermaid 绘制，调用 `mermaid_chart_generator` skill 生成
- [ ] **流程图可读性**：节点命名清晰,无歧义

### 4.3 核心数据模型检查

- [ ] **领域模型设计**：定义核心实体(DO)、值对象、聚合根
- [ ] **数据库表设计**：包含建表语句、索引、约束
- [ ] **数据字典完整**：每个字段都有注释和说明
- [ ] **Region 字段**：所有核心表必须包含 `region` 字段
- [ ] **必要字段完整**：包含 `id`, `region`, `create_by`, `update_by`, `ctime`, `utime`, `is_valid`
- [ ] **字段命名规范**：符合 `data_object_naming.md` 规范
- [ ] **索引设计合理**：高频查询字段有索引,包含 region 的复合索引

### 4.4 接口定义列表检查

- [ ] **接口清单完整**：列出所有新增/修改的接口
- [ ] **接口格式统一**：REST 接口和 RPC 接口都定义
- [ ] **接口参数详细**：每个接口的请求/响应参数都有说明
- [ ] **接口返回值规范**：使用 `BaseResultDTO<T>` 或 `BasePageResultDTO<T>`
- [ ] **接口参数对象化**：不直接使用基本类型作为参数
- [ ] **API artifact 版本**：若涉及对外 API 契约变更，已说明 `groupId:artifactId`、当前版本、目标 `*-SNAPSHOT` 版本、需要修改的
  `pom.xml` / version property，以及下游升级/回滚方案
- [ ] **API 文档注解**：包含 `@MethodDoc`, `@ParamDoc`, `@ResultDoc` 注解示例
- [ ] **错误码定义**：列出所有可能的错误码

### 4.5 i18n 检查

- [ ] **国际化范围明确**：说明哪些内容需要支持多语言
- [ ] **多语言资源文件**：定义多语言资源文件结构
- [ ] **语言切换逻辑**：说明如何根据 region/language 切换语言

### 4.6 非功能性设计检查

#### 4.6.1 性能设计

- [ ] **性能指标明确**：QPS、RT、并发量等指标
- [ ] **性能优化方案**：缓存、异步、批量等优化手段
- [ ] **性能测试计划**：说明如何进行性能测试

#### 4.6.2 安全设计

- [ ] **鉴权方案**：说明接口的鉴权方式
- [ ] **数据加密**：敏感数据的加密存储和传输方案
- [ ] **防刷防攻击**：限流、防重放等安全措施

#### 4.6.3 可用性设计

- [ ] **容错设计**：降级、熔断、限流方案
- [ ] **数据一致性**：分布式事务、最终一致性方案
- [ ] **故障恢复**：备份、恢复方案

#### 4.6.4 监控和告警

- [ ] **监控指标**：定义关键监控指标(业务+技术)
- [ ] **告警规则**：定义告警阈值和告警方式
- [ ] **日志规范**：定义日志级别和日志内容

#### 4.6.5 数据迁移(如适用)

- [ ] **迁移方案**：新旧数据的迁移策略
- [ ] **数据校验**：迁移后的数据一致性校验
- [ ] **回滚方案**：迁移失败的回滚方案

#### 4.6.7 资金安全监控(条件必填：涉及资金)

**如果需求涉及资金,必须包含**：

- [ ] **资金流水记录**：所有资金变动都有流水记录
- [ ] **对账机制**：定期对账,发现差异及时告警
- [ ] **异常监控**：异常金额、异常频率的实时监控
- [ ] **风控规则**：风险识别和风控拦截规则

#### 4.6.8 多机房配置及部署(条件必填：多机房)

**如果需求涉及多机房,必须包含**：

- [ ] **多机房部署架构图**：清晰展示各机房的部署情况
- [ ] **机房数据隔离**：说明数据是否按机房隔离
- [ ] **跨机房调用策略**：同机房优先 vs 跨机房容灾
- [ ] **配置差异管理**：说明各机房配置的差异点

## 六、测试方案检查(第5章)

### 必须项

- [ ] **测试范围明确**：单元测试、集成测试、回归测试范围
- [ ] **测试用例列表**：列出主要测试场景和测试用例
- [ ] **测试数据准备**：说明测试数据的准备方式
- [ ] **测试环境准备**：说明测试环境的搭建要求

### 推荐项

- [ ] **自动化测试**：单元测试覆盖率目标,自动化测试方案
- [ ] **性能测试**：性能测试场景和性能指标
- [ ] **压力测试**：压力测试方案和容量评估

---

## 七、发布方案检查(第6章)

### 6.1 灰度策略(必填)

- [ ] **灰度维度明确**：按用户、按地区、按比例等
- [ ] **灰度步骤清晰**：第一步灰度 X%,第二步灰度 Y%,逐步放量
- [ ] **灰度观察指标**：说明每个灰度阶段要观察哪些指标
- [ ] **灰度回滚条件**：明确什么情况下需要回滚

### 6.2 回滚方案(必填)

- [ ] **回滚触发条件**：明确什么情况下需要回滚
- [ ] **回滚操作步骤**：详细的回滚操作步骤
- [ ] **数据回滚方案**：如有数据变更,说明数据如何回滚
- [ ] **回滚验证方式**：回滚后如何验证系统恢复正常

### 6.3 监控及验证方案(必填)

- [ ] **监控指标列表**：列出所有需要监控的指标
- [ ] **验证步骤**：发布后的验证步骤(功能验证、数据验证、性能验证)
- [ ] **验证责任人**：明确谁负责验证
- [ ] **验证时间窗口**：验证需要多长时间

---

## 八、排期检查(第7章)

### 必须项

- [ ] **开发排期**：列出每个功能模块的开发时间
- [ ] **联调排期**：如涉及外部系统,列出联调时间
- [ ] **测试排期**：单元测试、集成测试、回归测试时间
- [ ] **发布排期**：灰度发布、全量发布时间点
- [ ] **关键里程碑**：标注关键节点和依赖关系

---

## 九、架构合规性检查

### 必须项：分层架构规范

- [ ] **严格分层**：API → UI → App → Domain → Infra 五层清晰
- [ ] **依赖无循环**：禁止下层依赖上层；允许领域层依赖外层 API 模块（本工程以外），禁止领域层依赖本工程 API 模块
- [ ] **数据对象命名**：DTO、VO、BO、DO、Entity 命名规范
- [ ] **接口与实现分离**：所有 Service、Repository、DAO 都定义接口

### 必须项：命名规范

- [ ] **类命名规范**：Service、AppService、DomainService、DAO、Mapper、Proxy 等后缀正确
- [ ] **数据对象后缀**：DTO、VO、BO、DO、Entity 使用正确
- [ ] **复数命名规范**：复数集合必须使用 `list` 后缀(如 `accountList`),禁止使用 `accounts`
- [ ] **参数对象命名**：QueryDO、Param、SyncParam 等后缀正确
- [ ] **业务术语标准**：使用标准英文命名,如 merchant(商家)、region(国家)、audit(审批) 等

### 必须项：数据库设计规范

- [ ] **表名规范**：小写字母,下划线分隔
- [ ] **字段名规范**：小写字母,下划线分隔,使用标准命名(如 create_by, update_by, ctime, utime)
- [ ] **必要字段**：id, region, create_by, update_by, ctime, utime, is_valid
- [ ] **主键设计**：bigint NOT NULL AUTO_INCREMENT
- [ ] **时间字段**：使用 bigint(20) 存储时间戳
- [ ] **Region 索引**：所有查询都必须包含 region 索引

### 必须项：MyBatis 规范

- [ ] **主键回填**：insert 方法配置 `@Options(useGeneratedKeys=true, keyProperty="id")`
- [ ] **字段完整性**：SqlProvider 包含所有实体类字段
- [ ] **Region 参数**：所有查询方法都包含 region 参数
- [ ] **分页规范**：使用 PageHelper,禁止手动 limit

## 十一、Mermaid 图表检查

- [ ] **图表可读性**：节点命名清晰,连线标注明确

## 使用方式

### 对于 AI Agent

1. **在设计阶段六完成后**：自动加载本检查清单
2. **逐项验证**：按照检查清单逐项验证设计文档
3. **生成报告**：生成质量检查报告,列出通过项和问题项
4. **修复问题**：如有问题,修复后重新检查
5. **用户确认**：向用户展示检查结果,请求最终确认

### 对于开发者

1. **在代码审查时**：使用本检查清单对设计文档进行人工审查
2. **在设计评审会上**：使用本检查清单作为评审依据
3. **在项目复盘时**：使用本检查清单评估设计质量
